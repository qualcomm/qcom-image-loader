1 Introduction

1.1 Purpose

This document describes the high-level design of the Qualcomm Image Loader (QIL) tool. It covers the system architecture, major components, protocol stack, data flows, external interfaces, and operational characteristics. It is intended to give architects, engineers, and QA teams a complete structural understanding of QIL without requiring a full review of the source code.

1.2 Scope

This document covers:

  • The layered software architecture of QIL
  • The design and responsibility of each major component
  • The SAHARA and Firehose protocol interaction model
  • The firmware flashing data flow from CLI invocation to device reset
  • The public SDK API surface
  • Firmware image validation subsystem
  • Platform abstraction and cross-platform support strategy
  • Build system, technology stack, and third-party dependencies
  • Operational concerns: logging, known limitations
  • Security considerations and licensing

This document does not describe low-level implementation details or source-code-level design. Those are covered by in-code documentation and the detailed design specifications for individual modules.

2 System Overview

2.1 Product Description

QIL is a command-line firmware flashing tool that transfers, manages, and validates firmware images on Qualcomm-based mobile devices. It communicates with devices operating in Emergency Download (EDL) mode — a low-level USB mode exposed by Qualcomm chipsets that bypasses the normal operating system boot path, providing direct access to the device's flash storage subsystem.

QIL addresses two categories of use:

  • Development use: Developers flash custom ROM builds, recovery images, and individual partitions during bring-up and validation cycles.
  • Manufacturing use: Factory tooling automates firmware provisioning and validation via the QIL SDK library integrated into host-side manufacturing test systems.

2.2 Distribution

QIL is distributed as a zipped package per supported platform, containing:

  • QIL executable (qil) — the standalone CLI binary
  • QIL User Guide — PDF user documentation
  • Qualcomm User Space Driver (optional) — pre-built USB driver for systems without Qualcomm kernel drivers

The source code is also available at https://github.com/qualcomm/qualcomm-image-loader.

3 Architecture

3.1 Layered Architecture

QIL is organised into six horizontal layers. Each layer depends only on the layer directly below it, producing a clean, testable separation of concerns. Higher layers are platform-agnostic; platform specificity is confined to the bottom two layers.

L1 · CLI Interface
main.cpp CliParser CliCommands Progress / Spinner UI --help / --version
↕
L2 · Public API
SoftwareDownload DeviceDiscovery ImageManagement
↕
L3 · Core Function
ImageTransfer Callback / Event Handlers Validation Engine Digest Logic
↕
L4 · Protocol Layer
SAHARA Protocol Firehose Protocol Firehose XML Loader RPC
↕
L5 · Communication
USB I/O Abstraction QDS Common Buffer / DataPacket Connection Manager
↕
L6 · Platform / HW
QDS Windows (qds_win) QDS Linux (qds_lnx) libusb-1.0 Platform Abstraction
Design Principle: Layers L1–L4 are fully platform-agnostic C++17. Platform-specific code is isolated entirely within L6, with L5 providing a portable interface over it. This makes porting QIL to new operating systems a matter of implementing a new L6 adapter only.

4 Component Design

4.1 CLI Layer (src/cli/)

The CLI layer is the topmost layer and serves as the sole entry point for end-users of the standalone executable.

main.cpp

Process entry point. Initialises the CLI parser, calls into CliCommands, handles top-level exception reporting, and sets the process exit code.

CliParser

Parses argc/argv into a structured command descriptor. Enforces required/optional argument rules per command. Emits formatted usage errors on bad input.

CliCommands

Dispatches the parsed command to the appropriate SDK API call. Translates SDK progress callbacks into terminal progress bars and status lines.

Progress / Spinner UI

Renders real-time transfer progress (percentage, speed, ETA) and operation spinners. Driven by event callbacks from the Core Function layer.

4.2 Public API Layer (src/exports/)

The Public API layer exposes a stable, versioned interface for host-tool integration without exposing internal implementation details. All three classes are the sole integration points for external consumers of the QIL static library.

SoftwareDownload

Top-level façade. Coordinates device discovery with image transfer operations. Accepts high-level flash parameters and routes them to ImageTransfer. Primary entry point for SDK consumers.

DeviceDiscovery

Enumerates EDL-mode devices on all connected USB buses. Supports synchronous polling and asynchronous hot-plug callbacks. Returns device identifiers (serial numbers) used by subsequent operations.

ImageManagement

Provides firmware operation primitives: flash build, erase partitions, read images, get flash info, and UFS provisioning. Each operation maps to a corresponding Firehose command sequence.

4.3 Core Function Layer (src/function/)

ImageTransfer is the central orchestration engine of QIL. It coordinates the two-stage SAHARA→Firehose protocol handoff, drives the partition write loop, manages the validation pipeline, and emits structured progress events throughout the lifecycle of a firmware operation.

Key responsibilities:

  • Transition device from SAHARA mode to Firehose mode by uploading the EDL programmer image
  • Parse the flat build directory to locate rawprogram.xml and patch.xml files
  • Iterate through each partition descriptor and issue the corresponding Firehose write command
  • Invoke the validation engine post-transfer based on the selected --validation-mode
  • Apply CDT image replacement when --cdt is specified
  • Issue the device reset command upon successful completion
  • Emit progress, error, and status events via registered callbacks to the CLI layer

4.4 Protocol Layer (src/protocol/)

The protocol layer implements the two EDL communication protocols used by Qualcomm devices. Both are binary/XML protocols transported over USB bulk transfer endpoints.

SAHARA Protocol Engine

Implements the Qualcomm SAHARA handshake and image-upload sequence. Responsible for sending the EDL programmer binary to the device and completing the transition to Firehose command mode. Handles Hello, Hello Response, Read Data, End Of Image Transfer, and Done packets.

Firehose Protocol Engine

Implements the full Firehose XML command-response protocol. Encodes partition configure, program, patch, erase, read, setbootablestoragedrive, and reset commands into XML payloads and parses device ACK/NAK responses.

Firehose XML Loader (firehose-loader/)

Parses rawprogram.xml and patch.xml command descriptor files using libxml2. Produces an ordered list of Firehose command objects consumed by the Firehose Protocol Engine. Supports multi-file semicolon-delimited lists.

RPC Layer (src/rpc/)

Provides a thin remote-procedure-call abstraction used internally for protocol command dispatch, enabling uniform encoding, timeout management, and retry semantics across both protocol engines.

4.5 Communication Layer (src/communication/)

The communication layer provides a portable, blocking USB I/O interface over the platform-specific QDS implementation. It abstracts bulk read/write operations, manages USB endpoint selection, and handles ZLP (Zero Length Packet) signalling. The --zlp-aware-host flag adjusts ZLP behaviour for WSL environments where usbipd-win changes the USB packet delivery model.

4.6 Device Management Layer (src/device/)

Device management tracks the full lifecycle of a connected EDL device from initial USB enumeration through flash completion.

  • Connection: Pairs a discovered device with a protocol handler and USB channel.
  • Manager (DeviceManager): Maintains the per-device state machine; handles connect, disconnect, and protocol transition events.
  • Buffer / DataPacket: Manages heap allocation for USB transfer buffers, enforcing alignment and size constraints required by the Firehose protocol.

5 Protocol Design

5.1 SAHARA Protocol

SAHARA (Secure Android Hardware Assisted Responsive Architecture) is a Qualcomm proprietary bootstrap protocol. When a device enters EDL mode it exposes USB interface 0x9008 and speaks SAHARA. The protocol's primary purpose in QIL is to upload the EDL programmer (a small Qualcomm-signed ELF binary) to the device's RAM, which then executes it and switches to Firehose mode.

SAHARA Packet Exchange Sequence

  1. Device sends Hello packet advertising its SAHARA version and mode
  2. Host responds with Hello Response, acknowledging version and selecting IMAGE_TX mode
  3. Device sends Read Data requests specifying byte offsets into the programmer image
  4. Host streams the requested programmer image segments
  5. Device sends End Of Image Transfer when programmer is fully received
  6. Host sends Done; device executes the programmer and switches to Firehose mode
--skip-sahara flag: When a device is already in Firehose mode (e.g., because a prior command used --reset=false or failed mid-transfer), the SAHARA stage can be bypassed with --skip-sahara to save time.

5.2 Firehose Protocol

Firehose is the main image-transfer protocol. It is an XML-over-USB protocol: the host sends XML command elements and the device responds with XML <response> or <log> elements. All binary image data is transferred as raw USB bulk payloads following the XML <program> handshake.

Stage 1 · SAHARA (Bootstrap)

  • Device powers on in EDL mode (USB 0x9008)
  • SAHARA Hello / Hello Response handshake
  • Host uploads EDL programmer binary
  • Device transitions to Firehose mode
⟶

Stage 2 · Firehose (Transfer)

  • Configure storage parameters
  • Parse rawprogram.xml / patch.xml
  • Erase target partitions (optional)
  • Stream firmware image data
  • Apply XML patch commands
  • UFS provision (if requested)
  • Validate and reset device

Key Firehose XML Commands Used by QIL

CommandDescription
<configure>Initialises the Firehose session; negotiates max packet size, memory type, and ZLP settings
<program>Writes a firmware image to specified LUN/partition/sector range
<patch>Applies a binary patch at a specific byte offset within a written partition
<erase>Erases a range of sectors on a specified LUN
<read>Reads sectors from the device back to the host (used for validation readback)
<getsha256digest>Requests the device to compute and return a SHA256 digest of a partition region
<setbootablestoragedrive>Sets the active bootable partition/slot index
<reset>Commands the device to reset and exit EDL mode
<getStorageInfo>Queries device storage geometry (LUN count, sector size, capacity)
<ufs-provision>Configures UFS logical unit parameters

6.2 VIP (Verified Image Programming) Download Flow

VIP download extends the standard flash flow with a cryptographic pre-authorisation step. A digest file is created offline from the firmware build, then passed alongside the build during flashing. The device verifies the digest before accepting the images.

  1. Offline (device not needed): Run --create-flash-build-vip-digest to produce a chained digest and signed digest pair from the flat build.
  2. Online: Run --flash-build with --chained-digest and --signed-digest pointing to the pre-created files. QIL transmits the digests to the device via Firehose before sending images; the device's secure boot subsystem validates them.
Important: Chained and signed digest files must be generated from the exact same firmware build that will be flashed. Any mismatch will cause the device to reject the image transfer.

8 Validation Subsystem

QIL implements a five-level validation framework selected at runtime via --validation-mode. Higher modes provide stronger integrity guarantees at the cost of additional transfer time (readback modes) or pre-computation effort (digest file modes).

ModeNameMechanismDigest File Required
0No ValidationNo post-flash verification; default if not specifiedNo
1Binary ReadbackRead back raw binary from device → compute digest₁ → compute digest₂ from original file at runtime → compareNo
2SHA256 ReadbackRead SHA256 digest directly from device → compute digest₂ from original file at runtime → compareNo
3Binary Readback + Digest FileRead back raw binary from device → compute digest₁ → load pre-created digest₂ from file → compareYes (--create-validation-digest)
4SHA256 Readback + Digest FileRead SHA256 digest directly from device → load pre-created digest₂ from file → compareYes (--create-validation-digest)

Modes 3 and 4 require a validation digest file to be pre-generated with --create-validation-digest before flashing. The digest file encodes expected hash values for each partition, enabling faster validation (no runtime digest computation from large image files) and offline authorisation of firmware builds.

When --read-image-path is provided alongside validation mode 1 or 3, the binary data read back from the device is saved to the specified directory for manual inspection.

The --validate-image-size flag adds a separate pre-transfer size check: it compares each firmware image file size against the partition size declared in rawprogram.xml and fails fast if any image exceeds the declared size, before any data is sent to the device.

9 Platform Abstraction

Cross-platform support is achieved through a two-part abstraction strategy:

  1. QDS (Qualcomm Device Service): The USB hardware access layer has two concrete implementations — qds_win and qds_lnx — that implement an identical interface declared in qds/common/. The build system selects the correct implementation at compile time via CMake platform guards.
  2. Platform Abstraction Layer (src/platform/): Wraps OS-level primitives (threading, timers, path handling) behind a uniform interface used throughout the L3–L5 layers.
PlatformUSB AccessDriver RequirementZLP Behaviour
Windows x64 / ARM64QDS Windows (qds_win)Qualcomm user-space driver or WDK WinUSBZLP-aware by default
Linux x86_64QDS Linux (qds_lnx)Kernel CDC/ACM driver; udev rules for permissionsZLP-aware by default
WSLQDS Linux (same binary)usbipd-win passthroughSet --zlp-aware-host=false

libusb-1.0 (statically linked) provides a portable USB bulk transfer API used by both QDS implementations for the actual byte-level I/O.

12 Operational Aspects

12.1 Logging

QIL writes structured debug logs to persistent directories automatically during operation. The --verbose flag additionally echoes detailed debug output to stdout.

PlatformLog Directory
Linux / WSL/var/tmp/QFS/QIL/Logs/

12.2 Known Limitations

#LimitationWorkaround
1 ADB and QIL cannot run simultaneously. Both tools contend for exclusive ownership of the USB device interface handle. Stop ADB before using QIL: adb kill-server
2 EDL mode required for all device operations. QIL cannot communicate with devices in normal Android boot mode. Use adb reboot edl or hardware volume-button sequence to enter EDL mode first.
3 Single device at a time. Parallel multi-device flashing is not supported in the CLI. Script sequential QIL invocations; or integrate via the SDK to manage device queuing in the host application.
4 WSL ZLP behaviour. usbipd-win changes USB packet delivery semantics relative to native Linux. Always pass --zlp-aware-host=false when running QIL inside WSL.

13 Security Considerations

  • EDL programmer authenticity: The EDL programmer binary uploaded during the SAHARA stage must be a Qualcomm-signed image. The device's secure boot ROM verifies the signature before executing the programmer. QIL does not bypass or modify this verification step.
  • VIP (Verified Image Programming): The VIP digest mechanism (--chained-digest / --signed-digest) allows the device's secure boot subsystem to cryptographically verify firmware builds before they are written to flash. Deployment environments with strict firmware integrity requirements should use VIP mode.
  • Physical access requirement: All QIL operations require physical USB connection to the target device in EDL mode. There is no network-accessible attack surface.
  • Log file sensitivity: QIL logs may contain device serial numbers, firmware build paths, and partition contents (in binary readback validation modes). Log directories should be protected with appropriate OS-level access controls in production environments.
  • libusb LGPL compliance: libusb is statically linked. See doc/LGPL-COMPLIANCE.md for redistribution obligations.
